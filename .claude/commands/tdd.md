# /tdd — 測試驅動開發

## 核心流程：RED → GREEN → REFACTOR

1. **定義介面** — 預期的輸入/輸出
2. **寫失敗測試（RED）** — 在實作之前先寫測試
3. **最小實作（GREEN）** — 只寫剛好能通過測試的程式碼
4. **重構（REFACTOR）** — 在測試通過的前提下改善程式碼
5. **驗證覆蓋率** — 確保 80%+ 覆蓋率

## 使用時機

- 開發新功能
- 新增函式或元件
- 修復 bug
- 重構程式碼
- 關鍵業務邏輯

## 關鍵實踐

### 必須做到
- 先寫測試，再寫實作
- 執行測試確認失敗，再開始寫程式碼
- 只寫剛好通過測試的程式碼
- 測試通過後才重構

### 避免的錯誤
- 在測試之前就寫實作
- 跳過步驟之間的測試執行
- 一次實作太多功能
- 測試實作細節而非行為

## 覆蓋率標準

- **一般程式碼**: 80%+
- **金額計算**: 100%
- **驗證邏輯**: 100%
- **核心業務邏輯**: 100%

- `/build-fix` — 修復建置問題
- `/code-review` — 審查實作結果
